# 04 - Bohemia / DayZ Modding Stack: Architecture

Current as of early 2026. Sources inline. Flag = uncertainty.

## Enforce Script

**What it is.** The OOP scripting language of the Enfusion engine, introduced with DayZ Standalone. Replaces SQF from the Arma series ([Bohemia Wiki - Enforce Script Syntax](https://community.bistudio.com/wiki/DayZ:Enforce_Script_Syntax)).

**Flavor.** C#-like. Strong typing, classes, inheritance, curly braces, explicit declarations (`int a; string s = "x";`), `auto` for type inference. Has `modded class Foo {}` syntax - the way you override vanilla behaviour.

**Execution model.** Interpreted bytecode on the Enfusion VM. Not a true JIT. Official docs don't publicly describe the internals - the community understanding is reverse-engineered ([gist: Arbitrary Enforce Execution](https://gist.github.com/wrdg/1893dd2e24faed933d6f643f8078fa78)). Flag: details here are community-lore, not Bohemia-documented.

**Module system.** Five modules load in order at startup: `1_Core`, `2_Gamelib`, `3_Game`, `4_World`, `5_Mission`. Higher can reference lower but not vice versa. Constants can be overridden at compile by the last-loaded mod -> **mod load order matters** (which is why Community Framework must always load first).

**Key gotchas** ([DayZ-Expansion EnforceScript Pitfalls wiki](https://github.com/salutesh/DayZ-Expansion-Scripts/wiki/%5BScripting%5D-DayZ-EnforceScript-Pitfalls)):
- **Garbage collection is aggressive.** Everything except `Managed`-derived classes is a weak reference. The GC destroys instances the moment they leave scope -> frequent segfaults if you forget to hold a `ref`.
- **Variable scoping.** Declaring `int x;` in both if and else branches throws a multiple-declaration error. Move it above.
- **Operator precedence surprises.** `a & b == b` parses as `a & (b == b)`.
- **`switch` default cases** sometimes still require an explicit return; workaround is returning outside the switch.
- **`GetType()` vs `ClassName()`.** Prefer `<instance>.GetType()` - returns the config.cpp class name. `ClassName()` is `Type().ToString()`.

**Debugging.**
- `Print()` and `PrintToRPT()` for logs.
- Workbench Script Editor includes an in-engine debugger.
- Community tooling: [VS Code Enforce extension](https://github.com/JeromeK13/vscode-dayz-enforce-script-extension), [enforce-script-lsp](https://github.com/devz-tools/enforce-script-lsp).
- Runtime errors often segfault rather than throw a clean script exception. Debugging is noticeably worse than modern scripting language ecosystems - closer to 2005-era C++ than to TypeScript.

## PBO Packaging

A PBO (Packed Bohemia Object) is the distribution unit. Build pipeline:

1. **P:\ drive setup.** DayZ Tools wants a P:\ drive (minimum ~20GB) for its workspace. Mount a folder as a drive letter.
2. **Extract game data** via Tools -> Extract Game Data. Re-extract after every major game update.
3. **Author** in `P:\Mods\@ModName\` with `addons/` and `keys/` subfolders.
4. **Binarize & pack** with **Addon Builder** - flags: `-binarize` (default, converts to engine format), `-packonly` (skip binarization), `-sign` (create .bisign), `-clear` (wipe temp) ([Bohemia Wiki - Addon Builder](https://community.bistudio.com/wiki/Addon_Builder)).
5. **Sign** with `.biprivatekey` (generated by DSCreateKey or DS utils from the Tools suite). Produces `.bisign` files.
6. **Test** with `-filePatching` launch parameter (allows loading unbinarized content for dev). Turn off for release.
7. **Publish** with Publisher (Steam Workshop upload).

**Community alternative:** [Mikero's pboProject](https://wiki.mikero.tools/tools/pboproject) - favored by ACE, CBA, CUP for batch builds, auto-sign, auto-deliver to `@mod` folders, better error checking.

**Signing is mandatory for public release** - `verifySignatures = 2` is the only correct production value in `serverDZ.cfg`.

Flag: "File Bank" - historically a Bohemia tool for raw PBO pack/unpack without binarization - isn't clearly documented in 2025 DayZ Tools. Addon Builder covers the same ground.

## Central Economy (CE)

Server-side loot/economy management. XML-driven, reloadable without game updates. Driven by files in the mission folder.

**Key XML files** ([DZconfig wiki](https://dzconfig.com/wiki/)):

- **`economy.xml`** - master switch. Each `<ce>` entity type (dynamic, animals, zombies, vehicles, randoms, custom, building, player) has `init` / `load` / `respawn` / `save` booleans. Lists which config files the economy reads.
- **`types.xml`** - loot config. Per-item:
  - `nominal` - target count. Server tops up toward this.
  - `min` - panic threshold. Server prioritizes respawning below this.
  - `lifetime` - seconds before cleanup; resets on player interaction.
  - `restock` - minimum respawn delay after pickup/destruction.
  - `quantmin` / `quantmax` - % fill for stackables/liquids; `-1` for non-quantity.
  - Flags: `count_in_map` (ground), `count_in_hoarder` (stashes/bases), `count_in_player` (carried).
  - `<category>` / `<usage>` / `<value>` tags drive what buildings can spawn it (e.g. Category="weapons" + Usage="Military" -> military buildings).
- **`events.xml`** - helicopter crashes, contaminated zones, animal/vehicle spawns. `active` / `position` (fixed or random) / gas-contamination flags. Naming prefixes matter: `Vehicle`, `Static`, `Item`.
- **`cfgspawnabletypes.xml`** - what spawns inside containers/events (heli-crash loot tables).
- **`cfgeconomycore.xml`** - CE core: dynamic event parameters, logging, backup system (Chernarus = 12 world segments), performance tuning.

**How it works.** ~10,000+ spawn positions per map. When `count_in_map` of an item drops below `min`, server spawns more toward `nominal`. Cleanup kicks at `lifetime`. Category + usage tags determine which buildings are eligible.

**Critical XML formatting.** A single missing closing tag kills CE init -> "loot drought," zero loot anywhere. Validate before deploying.

**Official reference:** [BohemiaInteractive/DayZ-Central-Economy](https://github.com/BohemiaInteractive/DayZ-Central-Economy).

## Client vs Server Mod Split

Three file types:

| File | Extension | Who has it | Purpose |
|---|---|---|---|
| Private key | `.biprivatekey` | You only | Signs PBOs |
| Public key | `.bikey` | Server's `keys/` | Trust anchor |
| Signature | `.bisign` | Everyone who has the PBO | Per-PBO signature |

**Signing workflow.**
- `DSCreateKey mymod` -> `mymod.biprivatekey` + `mymod.bikey`.
- Addon Builder `-sign` -> one `.bisign` per PBO.
- Distribute `.bikey` with mod; server admin drops it into `keys/`.
- `verifySignatures = 2` in serverDZ.cfg is the only correct production value.

**Key reuse.** You do NOT need a new key per mod. Reuse keys across mods so admins who trust you once trust you for all.

**Common failure mode.** Missing or mismatched `.bikey` -> server silently ignores the mod, client errors with "Client has a mod which is not on the server" ([Bohemia feedback T166329](https://feedback.bistudio.com/T166329)).

**Where code runs:**
- **Server-only:** persistence, economy, admin tools (Community Online Tools), anti-cheat, server events.
- **Client-only:** UI, HUD, cosmetic effects (rare because signing).
- **Both:** gameplay mods adding items/mechanics/vehicles. Most mods. **CF must always load first.**
- Use v2 signatures (v1 has been deprecated since Arma 2 v1.10).

## Hive / Persistence

**The modern truth:** There is **no third-party-accessible central hive**. The ARMA 2 / DayZ Mod era of a shared character DB is gone.

- **Official hive** = Bohemia-run, only reachable by Bohemia's own official servers. Zero community access.
- **Private shards** (every community server) = **local per-server** persistence.

**Local persistence files** in `mpmissions/<mission>/storage_1/`:
- **`players.db`** - SQLite. Columns: ID, ALIVE, UID, Data (BLOB with full player state+inventory). No official BLOB editor. Reverse-engineering it breaks ToS ([Steam SERVER FILES MEGATHREAD](https://steamcommunity.com/app/221100/discussions/0/1735465524703620457/)).
- **`*.bin`** files per entity category (dynamic_XXX.bin, vehicles.bin, events.bin, types.bin). Chernarus split into 12 world segments.
- **Backup system** configured via `cfgeconomycore.xml` (`backup` count, `backup_period` interval). `storageAutoFix = 1` replaces corrupted files with empty ones.

**Cross-server persistence is modded:** [**MapLink Hive**](https://steamcommunity.com/sharedfiles/filedetails/id=3478983598) is the primary mod (updated for 1.28 Feb 2025). Built on [Daemonforge's **UniversalAPI** webservice](https://github.com/DaemonForge/DayZ-UniveralApi) - this is the de-facto external-data backplane the scene has standardized on.

**Third-party DBs / external APIs - grey area:**
- No explicit Bohemia ban on external HTTP/DB integration.
- Tolerated: UniversalAPI (webservice for globals/player/object JSON storage), [CRDTN Core uses REST for shop + auth](https://dayz.foxapo.com/mods/crdtn-core/rest-api), various Discord bot bridges, DZconfig, CFTools Cloud.
- Not tolerated: using external APIs to **gate gameplay behind payment** -> runs into [Bohemia Server Monetization Rules](https://www.bohemia.net/monetization). The line between "technical integration" and "paywall enforcement" isn't codified. Operate conservatively.
- Flag: this is the single biggest policy uncertainty in the stack.

## RPC / Networking

Modders send messages between server and clients via RPCs.

**Native: `ScriptRPC`.** In `scripts/game/gameplay.c`. Example:
```enforce
GetGame().RPCSingleParam(player, ERPCs.RPC_USER_ACTION_MESSAGE, new Param1<string>(msg), true, player.GetIdentity());
```
Args: target, RPC ID enum, params, guaranteed-delivery bool, recipient identity.

**Community Framework's `RPCManager`** (de-facto standard):
- Resolves RPC ID collisions between mods.
- `GetRPCManager().AddRPC(modName, funcName, instance, execType)` - register.
- `GetRPCManager().SendRPC(modName, funcName, params, guaranteed, identity, target)` - send.
- NULL identity + server-side -> broadcast to all; NULL + client-side -> send to server.
- NetSync variables for continuous state sync (`RegisterNetSyncVariable` + `SetSynchDirty`).

**RF (RPC Framework)** - lighter alternative, created in response to CF's growing scope. [GitHub: mdcdesign/RF](https://github.com/mdcdesign/RF). Same API shape but adds **password-secured RPCs** (`SendSecureRPC` + `SetPassword`) that reject calls lacking a shared secret - stops rogue mods from invoking your RPCs. Logging via `-dorpclogs 1|2`.

**Which to use.**
- **CF** - most mods use it. Required if you integrate with Expansion, COT, Trader, etc.
- **RF** - if you want to avoid CF's footprint (ModStorage, notifications, etc.) and care about RPC-only security.
- **Native** - zero deps, manage IDs yourself. Rarely worth it.

Bandwidth: keep RPCs rare and small; validate server-side; mark critical ones `guaranteed=true`.

## Extensions (DLL-level)

C/C++/C# DLLs (.so on Linux) loaded by the game via the **RVExtension API** ([Bohemia Wiki - Extensions](https://community.bistudio.com/wiki/Extensions)). Entry points: `RVExtension`, `RVExtensionArgs`, `RVExtensionVersion`, `RVExtensionRegisterCallback`, `RVExtensionContext`.

**Bohemia's own warning:** _"Extensions can contain malicious code! Be sure to only use extensions from reliable developers."_ And: _"[Open source] will not protect you."_ They are serious about this.

**Client-side DLLs are blocked.** BattlEye blocks injected DLLs in the game process. Examples from feedback tracker:
- ReShade (`dxgi.dll`) -> blocked ([T184753](https://feedback.bistudio.com/T184753)).
- `version.dll`, `uxtheme.dll`, `edputil.dll`, `urlmon.dll`, `hid.dll` in system folders -> blocked, error `0xc000009a`.

**Server-side extensions are tolerated.** Server owners use C++ extensions for: database integration, external REST calls, perf-critical logic, Discord bridges, hive-access layers (legacy). Loaded by `DayZServer.exe`, not distributed through Workshop.

**Extensions are NOT on Steam Workshop.** Workshop is PBO-and-sig-verified. DLL distribution is manual - GitHub releases, Discord, direct download. This is a trust and distribution burden.

**Rule of thumb:** Stay in Enforce Script unless you absolutely need native (perf-critical or external I/O). Server-side extensions are where modders drop to C++.

## Workshop vs Direct Distribution

**Steam Workshop = sole official channel.** Players subscribe; DayZ Launcher auto-downloads to the `!Workshop` dir on next launch.

**Server operator flow** (the manual part):
1. Subscribe on Workshop.
2. Launch DayZ -> mod lands in `!Workshop/`.
3. Copy each `@ModName/` folder to the server root (next to `DayZServer.exe`).
4. Copy each mod's `.bikey` to server's `keys/` directory.
5. `-mod=@CF;@Expansion;@YourMod` in startup parameters. **CF must be first.**
6. Client connects -> signature verification -> match -> playable.

Automation: SteamCMD anonymous scripts + host-panel key management. Many GSPs now auto-sync keys ([Citadel](https://citadelservers.com/client/knowledgebase/404/DayZ-Server-How-To-Install-and-Update-Mods-in-Steam-Workshop.html)).

**Direct distribution** (GitHub release / Discord / private upload) bypasses Workshop. No auto-updates, higher trust cost, not Bohemia-supported for public mods. Used for:
- Beta/testing builds.
- Server-only mods that don't need client-side presence.
- DLL extensions.
- Mods that don't want to be on Workshop for DMCA / enforcement reasons.

## Server Hosting Reality

**GSP dominance:**
- **Nitrado** - the big one. Console exclusive (Xbox/PS5 server hosting is Nitrado-only). Full mod support, one-click Workshop install, daily backups, DDoS protection. Console ops complain there's only one performance tier and no real choice ([BI Feedback T178800](https://feedback.bistudio.com/T178800)).
- Alternatives (PC only): Game Host Bros, GameServers, RocketNode, Host Havoc, LightNode.
- Pricing: $12-22/mo entry, $30-80/mo for a 60-slot.

**Self-hosted.** Rent a VPS or use a dedicated box. 4GB RAM minimum, 8GB+ standard. SteamCMD anonymous download of app_id `223350`. Open UDP 2302+. For perf at scale: single-thread CPU matters more than core count (Enfusion is single-thread-dominant). Successful 2025 ops use Ryzen 9 7950X3D class CPUs with DDR5 ECC ([Nexus Games - Optimize DayZ Server](https://nexus-games.com/us/blog/how-to-optimize-dayz-server-performance/)).

**The real operational stack server ops actually run:**
- **CF-Tools / Omega Manager** - industry standard. SaaS web dashboard + shared banlist + real-time killfeeds + RCON. In-game requires CF + GameLabs mods. [CFTools docs](https://help.freakhosting.com/games/dayz/how-to-admin-your-server-with-cftools). Recent gripes (Sep 2025): Omega Manager Steam Authenticator issues, Architect tool broken on Win11.
- **Community Framework (CF)** - ~every script mod depends on it. [GitHub: Arkensor/DayZ-CommunityFramework](https://github.com/Arkensor/DayZ-CommunityFramework). Arkensor now appears affiliated with Bohemia ([BI Feedback tracker](https://feedback.bistudio.com/p/Arkensor/)).
- **Community Online Tools (COT)** - in-game GUI admin. Teleport, heal, weather, vehicle spawn. Modular so Expansion/HypeTrain plug in.
- **DayZ-Expansion** - the kitchen-sink gameplay overhaul. Helicopters, traders, missions, airdrops, territories, AI. [Discord: discord.gg/rMZuS4F](https://discord.gg/rMZuS4F).
- **DZconfig** - browser-based XML editor for types/events/traders/weather.
- **DayZ Server Manager (mr-guard)** - self-hosted NodeJS lifecycle manager with Discord bot.
- **VPP Admin Tools / ZomBerry** - in-game admin mods (alternative to COT).
- **DZM (DayZ Monitor)** - telemetry, heatmaps, session history, multi-layer anti-cheat.
- **DZloggy** - web log analytics (market/trader/ATM/territory/AI events).

A typical serious-community stack in 2026 looks like: **Nitrado** or dedicated Ryzen box -> **CFTools Cloud** for admin + banlist -> **CF + COT + Expansion + Trader + maybe AI + Namalsk** in the mod stack -> **DZconfig** for XMLs -> **Discord bot** for player-facing ops -> optionally **DZM + DZloggy** for telemetry.

## Bohemia's Rules & Enforcement

**[Server Monetization Rules](https://www.bohemia.net/monetization)** (DayZ - valid through 31 Jan 2027; Arma Reforger joined 2 Jun 2025):

Allowed only if you're registered on [bohemia.net/monetization/approved/dayz](https://www.bohemia.net/monetization/approved/dayz):
- Charging access (if fees/perks don't affect gameplay).
- Restricted-access paying-only shards (if they're on a different hive or have identical rules to free shards).
- Product placement / sponsorship / in-game advertising.
- Voluntary donations (no counter-value).
- Selling non-gameplay-affecting cosmetic perks.

Not allowed:
- Donations or rewards tied to in-game content (except crediting donors by name in docs).
- Selling anything that passed through BI Tools without express Bohemia authorization.
- Selling merch with Bohemia trademarks.
- Paywalling mods (despite this, paywalled mods exist on Workshop - enforcement is inconsistent).

Removal case studies:
- **Fidov mod / G65 mod** - removed for copyright infringement ([Steam discussion](https://steamcommunity.com/app/221100/discussions/0/3101264555057620575/)).
- **Terje Bruøygard's mods** - removed for bundling malicious DRM and concealed hacking/cheating tools ([Steam discussion](https://steamcommunity.com/app/221100/discussions/0/694250469803063413/ctp=2)).
- **DayZ Bounty** (historical, ARMA 2 era) - C&D'd for paying real money for in-game kills ([PC Gamer](https://www.pcgamer.com/dayz-bounty-cease-and-desist/)).

**DMCA abuse crisis (ongoing).** Bad actors have filed dozens of bogus DMCA takedowns against DayZ server operators to extort payment or harass competitors. Takedowns auto-pull mods for 10 days while counter-notice is processed; counter-notice requires disclosing your legal identity (real name + phone) to Steam ([Steam discussion](https://steamcommunity.com/discussions/forum/7/3758849893337016999/), [NotebookCheck](https://www.notebookcheck.net/DMCA-abuse-on-Steam-is-out-of-control-and-Valve-isn-t-fixing-it.1084373.0.html)). Some established modders now preemptively DMCA repackagers. If you ship anything popular, expect false DMCA to be part of your threat model.

**Third-party mod EULAs are real.** [RHS](https://www.rhsmods.org/page/EULA) explicitly forbids its assets being used in DayZ. Using content you don't have rights to is how you get pulled.

**Anti-cheat.** BattlEye banned ~20,000 DayZ cheaters in 2024 (down from ~30,000 in 2023). Server logs record movement/action/abnormality - server admins can detect speed hacks, teleports, item spawning.

**Arkensor -> Bohemia.** The maintainer of Community Framework is now responding to Bohemia feedback tracker tickets in a way that reads like internal testing access. Read: CF is now effectively semi-official infrastructure, not a pure community project. Good for stability, worth knowing politically.

**DayZ 2 trademark filed April 2026.** Bohemia filed with EUIPO ([Playfront](https://playfront.de/en/dayz-2-ist-offiziell-in-arbeit-bohemia-meldet-markenschutz-an/), [PCGamesN](https://www.pcgamesn.com/dayz/2-trademarked)). Successor game is in planning. DayZ 1.x modding has a finite runway; every mod you build now should be structured so you can port forward when the new Enfusion engine arrives.

## Summary: the mental model to carry forward

1. **Enforce is C#-lite on a quirky VM.** Garbage collection is the trap.
2. **PBO + bikey is the container.** CF-compatible, signed, Workshop-published.
3. **CE is XML-driven on the server.** Nothing in the game loop; it's config.
4. **Persistence is local.** Hive is Bohemia-internal. Cross-server is a mod.
5. **CF is effectively a platform.** You build on CF or you don't build.
6. **CFTools is the operator plane.** Not optional for serious servers.
7. **External APIs are tolerated but UN-gate-ed.** Stay technical, not monetized.
8. **DLLs are server-only in practice.** Clients are locked.
9. **Steam Workshop is the only legit distribution.** With DMCA-abuse risk baked in.
10. **Bohemia is watchful about monetization and malware.** Everything else is grey-area.
